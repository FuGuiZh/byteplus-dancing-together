# BytePlus Dancing Together Product Design System

## 设计上下文

目标用户是产品经理、研发和业务评审者。他们通过这个产品工作台判断「真人认证、素材入库、Seedance 2.0 生成视频」这条链路如何产品化，以及后续真实接入 API 时需要哪些状态、字段和兜底。

核心使用场景：

- 产品评审：理解用户从本人认证到生成舞蹈视频的体验。
- 研发评审：理解 `GroupId`、`AssetId`、endpoint、task id、payload 和状态机如何落到页面。
- API 接入态：已接入真实 BytePlus API 的链路直接请求上游；未接入或缺少凭证时返回本地 fallback 数据。

品牌气质：

- 高信任：真人、授权、审核、生成都要清楚可追踪。
- 工作台感：像一个真实产品后台，不像营销落地页。
- 克制但有质感：避免紫蓝霓虹、巨大渐变标题和堆卡片；用清晰排版、细线、状态色和少量动效建立高级感。

## 视觉方向

采用「清洁控制台 + 影像制作室」风格。

- 背景：偏暖白与浅灰，不使用纯白或纯黑。
- 主色：深墨色用于文本和导航。
- 强调色：单一青绿色，用于认证成功、可用素材、主要行动。
- 风险色：低饱和红色，用于失败、超时、合规提醒。
- 信息色：低饱和蓝色，用于排队、处理中、研发视角。

## 字体与排版

- 使用无衬线字体，优先 `Geist`、`Satoshi`、`Plus Jakarta Sans`，回退到系统 sans。
- 软件工作台不使用衬线字体。
- 标题控制克制，不做超大 hero。
- 数字、ID、payload 使用等宽字体。
- 页面主体信息密度中等，适合反复扫描。

## 布局规则

- 首屏就是可操作工作台，不做营销 hero。
- 桌面端采用三栏结构：
  - 左栏：流程导航和真人档案。
  - 中栏：当前工作台主任务。
  - 右栏：生成任务、研发视角、payload。
- 移动端折叠为单列，关键功能不能隐藏。
- 不嵌套卡片。模块可用边线、分隔线和留白组织。
- 卡片圆角不超过 8px，除非是状态胶囊或按钮。

## 组件规则

- 使用 `shadcn/ui` 风格组件，但必须定制颜色、圆角、阴影和状态。
- 图标使用 `lucide-react`。
- 按钮必须有层级：primary、secondary、ghost、danger。
- 表单 label 在 input 上方，错误提示在下方。
- 所有异步对象都必须有 loading、empty、error、success 状态。
- 状态标签颜色必须有语义，不用随机装饰色。

## 状态文案

文案要直接说明发生了什么和用户下一步能做什么。

- `Processing`：审核/处理中，不允许生成。
- `Active`：可使用，可加入生成任务。
- `Failed`：未通过，说明原因并提供重新上传。
- `queued`：排队中，可等待或取消。
- `running`：生成中，不承诺具体完成时间。
- `succeeded`：生成成功，并提示已转存。
- `expired`：任务超时，可重新提交。

## 研发视角

研发视角默认是辅助面板，不干扰产品主流程。必须展示：

- `GroupId`
- `AssetId`
- `asset://<AssetId>`
- `ProjectName`
- 普通版 / Fast endpoint
- 生成 task id
- 创建任务 payload
- 查询任务返回状态

## 动效规则

- 动效只用于状态变化、进度推进和面板展开。
- 使用 `transform` 和 `opacity`，避免布局抖动。
- 加载态使用骨架或进度步骤，不使用单独转圈作为主要反馈。
- 尊重 `prefers-reduced-motion`。

## 禁止事项

- 不使用紫蓝霓虹渐变作为主视觉。
- 不做纯营销落地页。
- 不把 `AssetId` 暴露给普通用户编辑 prompt。
- 不让用户选择未 `Active` 的素材生成。
- 不把 `.env.local`、AK/SK、API Key 写进页面或 Git。
